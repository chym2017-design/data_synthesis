# Synth Engine 用户操作与参数原理手册

本文档面向 Synth Engine 的业务使用者、数据配置人员和测试人员。它既说明页面如何操作，也说明每个主要参数影响什么、保存到哪里，以及系统内部如何把配置转换为合成数据。

本文档不包含任何真实 API Key、网页登录密码或 ECS 密码。

## 1. 系统用途

Synth Engine 用于生成银行 App 智能客服的意图识别测试与训练数据。系统的主要工作是：

1. 定义一级意图、子意图和示例；
2. 按概率生成虚拟用户画像；
3. 随机规划起始意图、目标意图、轮数、语气和心情；
4. 把规划结果和用户画像填入 Prompt；
5. 并行调用一个或多个 LLM 生成对话；
6. 过滤格式不合规的数据；
7. 导出 CSV 和 SFT JSONL；
8. 在合成前检查意图定义，在合成后检查重复、错标和模型一致性。

## 2. 访问与页面结构

当前部署入口：

```text
http://101.245.66.42/synth/
```

打开页面后，浏览器会先要求输入网站访问用户名和密码。账号和密码由管理员单独提供，本文档不保存密码。用户不能自行注册、创建密码或在前台修改密码；忘记密码时联系管理员重置。

每个用户名对应一个独立工作空间，名称格式为 `用户名-8位短标识`，例如 `zhangsan-a1b2c3d4`。页面中的模型配置、模板、上传文件、合成结果和质检结果都只属于当前用户，其他用户不能查看或修改。页面顶部会显示当前用户名、完整工作空间名称，以及全局运行/排队数量。

网页登录、服务器登录和模型调用使用三套不同的凭据：

| 凭据 | 使用者 | 用途 |
| --- | --- | --- |
| 网页用户名和网页密码 | 业务用户 | 打开 `/synth/` 页面 |
| ECS root/SSH 密码或 SSH 私钥 | 运维人员 | 登录 Ubuntu 服务器 |
| LLM/Embedding API Key | Synth Engine | 调用模型和向量接口 |

修改其中一套不会自动修改另外两套。例如，管理员修改 ECS 的 root 密码后，原网页密码仍然有效；修改网页密码也不会导致 API Key 失效。

当前入口使用 HTTP，网页账号、业务配置和返回内容没有 HTTPS 传输加密。不要在不可信酒店 Wi-Fi 中直接使用；应先连接可信 VPN，或者由运维人员配置 HTTPS。

左侧主要页面：

| 页面 | 用途 |
| --- | --- |
| 使用指南 | 查看首次使用准备、完整业务流程和各步骤入口 |
| 意图配置 | 管理一级意图、子意图、权重、描述和示例对话 |
| 对话配置 | 管理意图跳转、轮数、单轮比例、语气和心情 |
| 用户画像 | 管理人口属性、账户、持仓、贷款、行为和产品池 |
| Prompt 预览 | 检查 Prompt 模板变量是否正确填充 |
| 合成前质检 | 在合成前检查意图定义质量和语义混淆 |
| 数据合成 | 选择模板、设置基础样本数量和并行度并启动合成 |
| 合成后质检 | 检查合成数据的语义重复和意图标签一致性 |
| 任务队列 | 查看所有账号当前正在运行和等待执行的任务 |
| 配置管理 | 配置 LLM 和 Embedding API |
| 结果查看 | 预览、下载或删除自己工作空间内的合成与质检文件 |

页面右上角的“退出登录”会先要求确认，然后尝试清除浏览器保存的 Basic Auth 登录信息。Basic Auth 没有统一可靠的注销标准；如果退出后浏览器仍自动登录，请关闭该站点的全部窗口，或使用无痕窗口重新打开。

### 2.1 多用户与任务排队

- 管理员创建账号时，系统自动分配独立工作空间；
- 新空间会继承当前默认模型、模板和资源配置；
- 创建后的修改只影响自己的空间，不会覆盖其他用户；
- 合成和质检共用任务队列，服务器默认同时运行 5 个任务；
- 超出 5 个的任务会显示“排队中”、排队位置和全局等待数量；
- 所有账号都能查看全局队列中的任务 ID、用户名、类型和状态，但不能查看其他用户的配置或结果；
- 排队任务可以取消，已开始运行的任务当前不能安全强制中止；
- 服务重启会中断正在运行和排队的任务，需要重新提交。

“同时 5 个任务”是整台服务器的任务上限，不是每个用户各 5 个。任务内部的“并行度”是单个任务同时发出的模型请求数，两者不是同一个参数。

## 3. 推荐使用流程

首次使用建议按以下顺序操作：

1. 在“配置管理”中检查 LLM 和 Embedding；
2. 在“意图配置”中检查一级意图、子意图和示例；
3. 在“对话配置”中设置概率分布；
4. 在“用户画像”中设置画像概率和产品池；
5. 在“Prompt 预览”中确认没有未填充变量；
6. 运行“合成前质检”；
7. 先用 3～10 个基础样本试跑；
8. 检查结果格式后再扩大规模；
9. 运行“合成后质检”；
10. 在“结果查看”中下载最终文件。

不要一开始就运行大批量任务。LLM 配置、Prompt 或意图描述有误时，大批量任务会放大错误并消耗更多 API 额度。

### 4.1 基础配置与用户覆盖的关系

页面读取 YAML 配置时，系统先读取基础文件，再递归合并同名的 `.user.yaml`：

```text
最终配置 = 基础 YAML + 用户覆盖 YAML
```

覆盖文件只需要保存修改项，但当前页面通常会保存一份完整配置。相同字段以 `.user.yaml` 为准；没有覆盖的字段继续使用基础文件。

这样设计的目的，是在升级内置默认配置时尽量保留用户修改。`intent.json` 是例外：意图页面会直接更新当前用户工作空间中的 JSON，因为合成和质检也直接读取这个文件。

### 4.2 当前版本的页面保存与合成生效情况

当前版本不同配置的“页面显示、页面保存、正式合成”并不完全一致：

| 配置 | 页面保存位置 | 正式合成是否自动使用覆盖文件 | 当前注意事项 |
| --- | --- | --- | --- |
| LLM | `llm_config.yaml` | 是 | 直接覆盖全局配置 |
| Embedding | `embedding_config.yaml` | 是 | 直接覆盖全局配置 |
| 意图 | `intent.json` | 是 | 页面、合成和质检使用同一份 JSON |
| 对话 | `synth_config.user.yaml` | 否 | 前端启动任务未传 `synth_config_override`，流水线默认读取基础 YAML |
| 画像 | `profile_config.user.yaml` | 是 | 正式合成会递归合并基础配置和用户覆盖 |

这意味着页面提示“保存成功”只代表文件写入成功，不一定代表当前合成流水线已经使用了该覆盖文件。

#### 意图配置

“意图配置”页面保存时会直接写入当前用户工作空间的：

```text
intent.json
```

保存采用临时文件替换，成功后新提交的合成和质检任务会读取修改后的内容。旧版本遗留的 `intent.user.yaml` 会在下次保存时清除，避免它覆盖新的 JSON。已经开始运行的任务不会中途重新加载意图配置。

#### 对话配置

“对话配置”页面保存到 `synth_config.user.yaml`，通用配置读取接口会显示合并后的结果，但当前“合成任务”页面提交的请求只包含模板、样本数量和并行度，没有提交 `synth_config_override`。后端在没有覆盖参数时直接读取基础 `synth_config.yaml`。

因此当前版本要让对话参数实际影响合成，需要把确认后的参数同步到基础 `synth_config.yaml`，或者由开发人员修复启动参数/后端合并逻辑。

#### 画像配置

正式合成会正确合并 `profile_config.yaml` 和 `profile_config.user.yaml`。但是“随机生成示例画像”接口当前只读取基础 `profile_config.yaml`，所以页面示例可能与正式合成使用的画像覆盖配置不完全一致。

## 5. 配置管理：LLM

每个模型卡片都有“测试连接”按钮。它会使用当前页面中尚未保存的模型名称、API 地址和 API Key 发起一次短文本非流式请求，并显示是否成功、耗时和回复摘要。建议按“填写 → 测试连接 → 保存”的顺序操作。测试会真实调用模型接口，可能产生少量费用。

### 5.1 操作方法

1. 打开“配置管理”；
2. 在 LLM 区域添加或编辑模型；
3. 填写模型名称、API 地址和 API Key；
4. 设置超时、最大输出长度、温度等参数；
5. 点击保存；
6. 用小规模合成任务验证。

可以添加多个模型。合成时，系统会按请求序号轮询分配模型，实现简单负载均衡。

### 5.2 LLM 参数说明

| 页面参数 | YAML 字段 | 含义 | 常见建议 |
| --- | --- | --- | --- |
| 模型名称 | `model` | 发送给 API 的模型标识，也是日志中的模型名 | 必须与供应商实际模型 ID 一致 |
| API 地址 | `url` | Chat Completions 请求地址 | 通常应包含 `/chat/completions` |
| API Key | `api_key` | 调用模型的身份凭据 | 不要提交到公开仓库 |
| 温度 | `temperature` | 控制随机性和多样性 | 数据合成常用 `0.7～1.5`；越高越发散 |
| 最大 Token | `max_tokens` | 单次模型输出长度上限 | 太小会截断多轮对话；太大增加成本 |
| 超时 | `timeout` | 单次请求最长等待秒数 | 网络较慢时可适当增大 |
| 流式输出 | `stream` | 是否使用 SSE 流式接收 | 取决于供应商兼容性 |
| 思考模式 | `enable_thinking` | 是否请求模型启用推理/思考能力 | 复杂判断可能更稳，但更慢、更贵 |

配置文件：

```text
/opt/synth_engine/workspaces/<workspace_id>/configs/llm_config.yaml
```

### 5.3 多模型与并行原理

“多个模型”和“并行度”不是同一个概念：

- 多个模型：请求按顺序轮询分配给不同配置；
- 并行度：线程池中最多同时运行多少个请求；
- 重试：请求异常或状态码不是 200 时，单个请求最多自动尝试 5 次；
- 并行度不会把样本数量乘倍。

并行度过高可能触发供应商限流、连接超时或账户额度限制。一般从 `2～3` 开始，根据 API 限额调整。

## 6. 配置管理：Embedding

Embedding 用于把文本转换为向量，主要服务于相似度质检。

“测试连接”会请求一个短文本向量，成功时显示向量维度。若失败，重点检查 API 地址、模型名、API Key 和网络权限。测试同样可能产生少量费用。

### 6.1 参数说明

| 页面参数 | YAML 字段 | 含义 |
| --- | --- | --- |
| 模型 | `model` | Embedding 模型名称，如 `BAAI/bge-m3` |
| API 地址 | `url` | OpenAI 兼容 API 的基础地址 |
| API Key | `api_key` | Embedding 服务访问密钥 |

配置文件：

```text
/opt/synth_engine/workspaces/<workspace_id>/configs/embedding_config.yaml
```

Embedding 配置错误时，合成任务仍可能运行，但合成前/后相似度质检会失败。

模型测试会直接显示服务商返回的错误，例如 `401` 表示密钥无效、`403` 可能表示余额不足或无权限、`404` 通常表示地址或模型不匹配。当前默认 SiliconFlow 配置若返回 `code 30001 / account balance is insufficient`，需要为该 API Key 所属账户充值或更换有余额的 Key，修改代码无法绕过服务商余额限制。

## 7. 意图配置

### 7.1 操作方法

1. 打开“意图配置”；
2. 展开一个一级意图；
3. 编辑描述、权重和“真子意图”开关；
4. 新增或修改子意图名称及描述；
5. 为子意图添加示例对话；
6. 点击“保存所有配置”；
7. 保存后执行合成前质检。

页面支持导出 JSON、粘贴完整 `intent.json` 或纯 `intent` 对象。

“恢复默认”会使用系统内置 `intent.json` 覆盖当前用户的全部意图修改。操作前会二次确认，执行后不能撤销；需要保留的意图应先导出 JSON。

### 7.2 参数说明

| 字段 | 含义 | 对合成的影响 |
| --- | --- | --- |
| 一级意图名称 | 业务大类名称 | 作为一级分类标签和 Prompt 变量 |
| `description` | 一级意图边界说明 | 帮助模型理解该意图包含和排除什么 |
| `weight` | 被抽中的相对权重 | 概率为“本意图权重 / 所有意图权重总和” |
| `true_sub_intent` | 子意图是否作为独立跳转对象 | 影响子意图在意图转换中的使用方式 |
| `sub_intent[].name` | 子意图名称 | 作为二级标签和目标标签 |
| `sub_intent[].description` | 子意图业务边界 | 直接进入 Prompt，描述越清楚越不易错标 |
| `example[].子意图` | 示例关联的子意图 | 说明示例属于哪个子意图 |
| `example[].示例` | 示例对话文本 | 为模型和质检提供语义参考 |

### 7.3 权重原理

权重不是百分比。假设三个一级意图权重分别为 `1、2、1`，则理论抽样概率约为：

```text
25%、50%、25%
```

权重为 0 的意图原则上不会被抽中。修改权重后应使用足够大的样本量观察分布，小样本会因随机性偏离理论比例。

### 7.4 描述编写建议

一个可用的子意图描述应同时说明：

- 用户真正想完成什么；
- 与相邻子意图的差异；
- 哪些表达不属于本意图；
- 是否依赖用户个人信息；
- 必要时给出反例。

只写一个短名称，会增加意图混淆和质检失败率。

## 8. 对话配置

页面保存的配置对应：

```text
/opt/synth_engine/workspaces/<workspace_id>/templates/bank_intent/synth_config.user.yaml
```

基础默认值对应：

```text
/opt/synth_engine/workspaces/<workspace_id>/templates/bank_intent/synth_config.yaml
```

当前版本正式合成不会自动使用页面保存的 `synth_config.user.yaml`，具体原因和处理方式见 4.2 节。以下参数说明仍然代表这些字段在合成引擎中的真实含义。

“恢复默认”会恢复内置 `synth_config.yaml` 并删除当前用户的覆盖配置。操作前会二次确认，执行后不能撤销。

### 8.1 意图跳转参数

| 页面参数 | YAML 字段 | 当前默认值 | 含义 |
| --- | --- | ---: | --- |
| 单意图概率 | `intent_transition.single_intent_prob` | `0.30` | 起始意图和目标意图相同，不发生意图跳转 |
| 多意图概率 | `intent_transition.multi_intent_prob` | `0.70` | 对话包含意图变化；页面与单意图概率联动 |
| 同意图多子意图 | `same_intent_multi_subintent_ratio` | `0.80` | 已进入多意图分支后，在同一一级意图内切换子意图的条件概率 |
| 跨意图跳转 | `cross_intent_jump` | `0.20` | 已进入多意图分支后，切换到另一个一级意图的条件概率 |

完整概率关系：

```text
同一级意图内换子意图的总概率
= multi_intent_prob × same_intent_multi_subintent_ratio

跨一级意图跳转的总概率
= multi_intent_prob × cross_intent_jump
```

页面会让两组互补参数分别保持和为 1。

### 8.2 对话轮数分布

| YAML 字段 | 使用场景 |
| --- | --- |
| `round_distribution.single_intent` | 起始意图和目标意图相同 |
| `round_distribution.multi_intent` | 起始意图和目标意图不同 |

配置格式：

```yaml
- [轮数, 概率]
```

例如：

```yaml
- [2, 0.25]
- [3, 0.50]
```

表示该分支有 25% 概率规划为 2 轮，50% 概率规划为 3 轮。每一组概率建议归一化为 1；页面提供“概率重分布”。

一轮定义为一条用户消息加一条客服消息，因此：

```text
num_sen = 2 × round
```

### 8.3 单轮对话比例

| 页面参数 | YAML 字段 | 默认值 |
| --- | --- | ---: |
| 单轮对话比例 | `single_round_ratio` | `0.05` |

实现中单轮基础样本数为：

```text
max(1, round(基础样本数量 × single_round_ratio))
```

所以即使基础样本数量很小，系统也至少会安排 1 个单轮基础样本。

单轮 Prompt 默认一次要求模型生成 5 条用户请求，随后后端将每个以“用户：”开头的行拆成独立样本。因此“基础样本数量”不等于最终导出行数。

### 8.4 语气分布

| 默认语气 | 默认概率 | 含义 |
| --- | ---: | --- |
| 祈使句 | `0.60` | 请求或命令式表达 |
| 问句 | `0.15` | 疑问式表达 |
| 反问句 | `0.10` | 反问形式表达态度 |
| 倒装句 | `0.05` | 使用非标准语序增加多样性 |
| 长短复合句 | `0.05` | 生成较长、含多个分句的请求 |
| 带错别字的请求 | `0.05` | 模拟真实输入错误 |

字段位置：

```text
tone_distribution
```

语气概率总和必须为 1，否则后端配置验证可能失败。页面虽然允许暂时保存非 1 的显示状态，但正式运行前应点击“概率重分布”。

### 8.5 心情参数

| 页面参数 | YAML 字段 | 默认值/说明 |
| --- | --- | --- |
| 心情触发概率 | `mood.probability` | 默认 `0.30` |
| 情绪词池 | `mood.emotions` | 兴奋、开心、平静、焦虑等 |

命中心情概率后，系统会把语气扩展为类似：

```text
祈使句，用户当前的心理状态为焦虑
```

这段文本会进入 Prompt，影响模型生成风格。

### 8.6 终止轮切换比例

| 页面参数 | YAML 字段 | 默认值 |
| --- | --- | ---: |
| 终止轮切换比例 | `intent_transition_constraint.switch_at_terminal_turn_ratio` | `0.20` |

当起始子意图与目标子意图不一致，并命中该概率时，系统向 Prompt 注入时序约束：前面轮次保持起始意图，只允许最后一轮切换到目标意图。

### 9.5 概率、权重和范围的区别

- 概率：通常是 `0～1`，如 `0.7` 表示约 70%；

- 范围：通常为 `[最小值, 最大值]`，在范围内随机抽样；
- 数量上限：可能还会受到年龄、风险等级或候选池大小限制。

## 11. 合成前质检

### 11.1 作用

合成前质检针对 `intent.json`，目的是在消耗大量 LLM 额度前发现：

- 菜单/意图名称不清楚；
- 描述与子意图不一致；
- 示例质量差；
- 两个意图语义高度相似、边界混淆。

### 11.2 页面参数

## 12. 数据合成

### 12.1 操作方法

1. 打开“数据合成”；
2. 选择模板，当前主要模板为 `bank_intent`；
3. 填写“样本数量”；
4. 填写“并行度”；
5. 点击“启动合成”；
6. 等待状态依次经过生成、调用、过滤、导出和完成；
7. 到“结果查看”检查文件。

### 12.2 运行参数

| 页面参数 | API 字段 | 默认值 | 实际含义 |
| --- | --- | ---: | --- |
| 模板 | `template_name` | 必选 | 选择意图、画像、Prompt 和导出实现 |
| 样本数量 | `num_samples` | `20` | 范围 1～20；基础样本计划数，不是最终 CSV 行数硬上限 |
| 并行度 | `para` | `3` | 范围 1～5；单个任务的 LLM 并发请求数 |

### 12.3 为什么填写 3 可能得到 7 条

以一次真实运行逻辑为例：

1. `num_samples=3` 生成 3 个基础样本；
2. 系统至少把其中 1 个标为单轮；
3. 单轮 Prompt 要求模型一次生成 5 条请求；
4. 后端把 5 个“用户：...”拆成 5 条独立记录；
5. 如果另外 2 个是多轮，则最终为 `2 + 5 = 7`。

并行度 3 只表示同时处理 3 个请求，不表示生成 3 倍数据。

### 12.4 内部执行阶段

```text
基础样本计划
  → 用户画像生成
  → 意图对与轮数抽样
  → Prompt 填充
  → LLM 并行调用
  → 单轮拆分
  → 格式过滤
  → Query 提取
  → CSV/JSONL 导出
```

### 12.5 格式过滤规则

银行意图模板要求模型输出的每个非空行都必须以以下任一前缀开始：

```text
用户：
客服：
```

出现解释、标题、Markdown 说明或其他前缀时，整条样本可能被过滤。

## 14. 合成后质检

### 14.1 操作方法

1. 打开“合成后质检”；
2. 选择某次合成的 `data.csv`；
3. 选择对应的意图配置；
4. 设置采样数量、并行度和相似度阈值；
5. 决定是否跳过 Embedding 或 LLM 质检；
6. 启动任务；
7. 查看汇总、详细结果和相似样本对。

### 14.2 页面参数

| 参数 | API 字段 | 默认值 | 含义 |
| --- | --- | ---: | --- |
| 数据文件 | `data_file` | 必选 | 被检查的合成 CSV |
| 意图文件 | `intent_file` | 必选 | 判断标签边界使用的意图定义 |
| 跳过 Embedding | `skip_embedding` | `false` | 不执行相似语料检查 |
| 跳过 LLM | `skip_llm` | `false` | 不执行多模型意图判定 |
| 并行度 | `para` | `3` | 范围 1～5；LLM 质检并发数 |
| 采样数量 | `sample_size` | `20` | 范围 1～20；最多检查 20 条 |
| 相似度阈值 | `similarity_threshold` | `0.95` | 只输出达到阈值的相似样本对 |

数据量较大时，两两相似度比较和多模型投票成本较高。建议先设置采样数量试跑。

### 14.3 Embedding 相似度原理

1. 对每条 `query` 计算向量；
2. 两两计算余弦相似度；
3. 只保留 `similarity >= threshold` 的样本对；
4. 如果两条文本语义接近但意图或子意图标签不同，则标记为潜在问题。

阈值接近 1 时更严格。`0.95` 主要用于寻找几乎重复但标签不一致的数据。

### 14.4 LLM 多模型投票原理

系统对每条样本最多选择 3 个已配置模型：

1. 各模型独立判断一级意图；
2. 一级判断正确时再判断二级子意图；
3. 正确票数达到一半或以上时判定通过；
4. 一级和二级都通过才是总体通过。

当前实现使用的判断条件是：

```text
正确票数 >= 模型数 / 2
```

因此配置 2 个模型时，1 票正确即可能满足该条件。这是当前代码行为，若业务需要严格多数，应由开发人员将规则调整为“正确票数 > 模型数 / 2”。

### 14.5 输出文件

| 文件 | 内容 |
| --- | --- |
| `embedding_similarity_check.csv` | 高相似度且标签可能冲突的文本对 |
| `qc_detail_by_model.csv` | 每个模型对每条样本的独立判断和完整 Prompt |
| `qc_summary_voting.csv` | 每条样本的投票汇总、通过状态和失败原因 |
| `meta.json` | 本次质检参数、状态和文件路径 |

## 17. 实现原理总览

### 17.1 配置驱动

系统尽量把业务变化放在 JSON/YAML/Markdown 中，而不是写死在 Python：

- 意图边界放在 `intent.json`；
- 对话概率放在 `synth_config.yaml`；
- 画像分布放在 `profile_config.yaml`；
- 生成规则放在 Prompt Markdown；
- Python 负责采样、编排、调用、过滤和导出。

### 17.2 随机性来源

最终数据的差异来自多个独立随机过程：

- 一级意图权重采样；
- 子意图选择；
- 意图跳转分支；
- 单轮/多轮分配；
- 对话轮数；
- 语气和心情；
- 用户画像各字段；
- LLM 温度和模型自身随机性。

因此同样参数重复运行，不保证生成完全相同的数据。

### 17.3 画像与意图的结合

画像先生成完整结构，再格式化为简化 Markdown 注入 Prompt。模型会看到年龄、职业、账户、持仓、贷款和资产摘要等信息，因此生成内容可能引用个人余额、产品或业务状态。

### 17.4 标签来源

标签不是模型生成后再猜测出来的，而是在调用 LLM 前由采样器预先确定：

```text
采样目标意图 → 让 LLM 按目标生成文本 → 保存预设标签
```

合成后质检的作用，就是重新让其他模型判断文本，检查生成文本是否真的符合预设标签。

## 18. 常见问题

### 18.1 页面保存成功，但合成结果没变化

依次检查：

1. 是否点击了“保存所有配置”；
2. 是否启动了新任务而不是查看旧结果；
3. 意图修改检查 `intent.json`，其他模板参数检查相应的 `.user.yaml`；
4. 修改的是不是意图或对话配置；
5. 对话配置是否遇到当前版本不自动应用 `synth_config.user.yaml` 的问题；
6. 若查看画像示例，是否遇到示例接口不合并画像覆盖文件的问题；
7. 小样本随机波动是否掩盖了概率变化。

### 18.2 最终数量比填写数量多

这是单轮基础样本拆分造成的当前设计行为，不是并发重复执行。

### 18.3 最终数量比填写数量少

可能原因：

- LLM 请求重试后仍失败；
- 输出包含解释或标题；
- 某些行没有以“用户：”或“客服：”开头；
- 输出被截断；
- 格式校验过滤了不合规样本。

检查对应任务的 `llm_log.jsonl`、`data.csv` 和应用日志。

### 18.4 生成内容总是很相似

可以检查：

- LLM 温度是否过低；
- 意图描述和示例是否过于单一；
- 语气分布是否集中在一个选项；
- 用户画像候选池是否过小；
- 是否只配置了一个模型；
- Prompt 是否限制过强。

### 18.5 生成内容偏离标签

可以检查：

- 子意图描述是否明确；
- 相邻子意图是否重叠；
- 示例是否错误关联；
- 跨意图概率是否过高；
- 终止轮切换约束是否需要提高；
- LLM 温度是否过高；
- Prompt 中目标变量是否正确填充。

### 18.6 任务长时间停在调用中

可能原因：API 限流、网络慢、超时设置过大、并发过高或模型流式协议不兼容。先降低并行度，并检查模型 API 日志和 `docker logs`。

### 18.7 502 Bad Gateway

这不是业务参数问题。它表示 Nginx 正常，但后端容器没有响应。需要运维人员检查 `synth-engine` 服务、Docker 容器和 `127.0.0.1:8080`。

## 19. 数据与安全注意事项

1. 页面中的用户画像是虚拟数据，但产品池、Prompt 和 API Key 仍可能属于内部信息；
2. 不要把真实客户身份证、卡号、手机号或交易数据上传到本系统；
3. API Key 只应保存在受控服务器配置中；
4. 当前 IP 入口使用 HTTP，不应在不可信网络中传输敏感配置；
5. 删除运行结果前先下载或备份；
6. 大规模任务前确认模型账户余额和限流策略；
7. 修改基础 JSON/YAML 前保留备份；
8. 不要同时由多人编辑同一配置页面，后保存者可能覆盖先保存者；
9. 不要在任务运行过程中替换 Prompt、意图或产品池文件。

## 20. 用户与开发/运维的职责边界

普通用户可以通过页面完成：

- LLM/Embedding 参数填写；
- 意图、对话和画像配置；
- Prompt 检查；
- 合成和质检任务；
- 结果预览与下载。

以下操作建议交给开发或运维人员：

- 修改最终样本数量的严格限制逻辑；
- 修改投票通过算法；
- 直接编辑服务器文件；
- 恢复被删除的运行结果；
- Docker/Nginx/systemd 故障处理；
- API Key 泄露后的轮换；
- HTTPS、域名、网络和服务器安全配置。
